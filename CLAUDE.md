# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

**Card Sorter** is a physical robot system that automates MTG card collection management. The system identifies cards via computer vision, tracks them in a MySQL database, and sorts them by value. It consists of four main components: a Go backend service, a Python/Kivy robot control interface, ESP32 arm firmware, and a React web frontend.

## Architecture

### System Components

**Backend** (`backend/`): Go service providing gRPC/HTTP APIs for card and library management. Uses sqlc for type-safe database queries, integrates with Scryfall API for card data, and handles user authentication via JWT. Detailed documentation in `backend/CLAUDE.md`.

**Robot Software** (`robot/software/`): Python/Kivy touchscreen application running on the robot. Handles camera capture, communicates with backend via gRPC, manages user sessions, and controls the sorting workflow through multiple screens (login, library selection, catalog, card results).

**Robot Arm** (`robot/arm/`): ESP32 Arduino firmware (PlatformIO) that controls the physical MaxArm robot arm with servo motors and suction nozzle for card manipulation.

**Web Frontend** (`web/frontend/`): React + TypeScript + Vite application providing web-based access to card collections and library management.

### Data Flow

1. User logs into robot touchscreen → JWT token stored locally
2. Robot captures card image → sends to backend for identification
3. Backend queries local SQLite database (card hashes) → returns card details
4. User confirms/edits card → backend updates user's library in MySQL
5. Robot arm sorts card into appropriate bin (valuable/identified/unknown)
6. Web frontend provides alternative interface to view/manage collections

### Database Architecture

**Remote MySQL** (`backend/`): Stores user data, libraries, card collections, Scryfall bulk data, and pricing information. Schema managed via golang-migrate, queries generated by sqlc.

**Local SQLite** (`cards.sqlite`): Card image hashes and local paths used by robot software for fast card identification. Populated by downloader script from Scryfall bulk data.

## Development Commands

### Backend (Go)

See `backend/CLAUDE.md` for detailed commands.

```bash
# Build backend service (from project root)
make backend

# Build CLI tool
cd backend && go build -o cardctl.exe ./cmd/cardctl

# Run database migrations
cd backend && make migrate

# Regenerate Go code from SQL queries (after editing internal/db/queries/*.sql)
cd backend && sqlc generate

# Update Scryfall data
cd backend && ./cardctl.exe scryfall update

# Deploy to production server
cd backend && make deploy
```

**After modifying protobuf definitions** (in `protos/` directory):
```bash
# From project root - regenerates Go, Python, and TypeScript clients
make protos

# Then rebuild backend
make backend
```

### Robot Software (Python)

The robot control application uses Python 3 with Kivy for the touchscreen UI.

```bash
cd robot/software

# Install dependencies (first time)
pip install -r requirements.txt

# Run the application
python main.py

# Deploy to robot device
make deploy

# Fetch captured card images from robot
make fetch-last
```

**Configuration**: Create `.env` file with:
- `CARDSORTER_BACKEND_HOST` - Backend server hostname/IP
- `CARDSORTER_BACKEND_PORT` - gRPC port (default 9090)

### Robot Arm (PlatformIO)

ESP32 firmware for physical robot arm control.

```bash
cd robot/arm

# Build firmware
pio run

# Upload to ESP32
pio run --target upload

# Monitor serial output
pio device monitor
```

**Dependencies**: MaxArm and Position libraries (defined in platformio.ini)

### Web Frontend (React)

```bash
cd web/frontend

# Install dependencies
npm install

# Generate API client from backend OpenAPI spec
npm run generate-client

# Development server
npm run dev

# Build for production
npm run build

# Lint code
npm run lint

# Preview production build
npm run preview
```

**Important**: After updating backend protobuf definitions, regenerate the OpenAPI spec in the backend, then run `npm run generate-client` to update the frontend client.

### Protocol Buffers

gRPC service definitions are in `protos/` directory. The `buf` tool generates:
- Go server code → `backend/gen/protos/`
- Python client code → `robot/software/magic_client/`
- OpenAPI spec → `backend/gen/openapiv2/openapi.swagger.json`

```bash
# Generate all protobuf code (run from project root)
make protos
```

**Important**: After running `make protos`, you must:
1. Rebuild the backend: `cd backend && go build -o cardsorter.exe .`
2. Regenerate frontend client: `cd web/frontend && npm run generate-client`

**Configuration**:
- `backend/.env` - Backend database credentials and JWT secret

**Data Files**:
- `cards.sqlite` - Local card database with image hashes (robot software)

## Multi-Component Workflows

### Adding a New API Endpoint

1. **Define in protobuf**: Edit `protos/*.proto` files
2. **Regenerate all code**: Run `make protos` from project root (regenerates Go, Python, TypeScript)
3. **Rebuild backend**: Run `make backend` from project root
4. **Implement server logic**: Edit `backend/internal/server/*_server.go`
5. **Update database** (if needed):
   - Add migration in `backend/internal/db/schema/`
   - Run `cd backend && make migrate`
   - Update queries in `backend/internal/db/queries/*.sql`
   - Regenerate query code: `cd backend && sqlc generate`
   - Rebuild: `make backend` from project root
6. **Test the endpoint** with the backend running locally

### Updating Card Database

**Backend**: Run `cardctl scryfall update` to download latest Scryfall data and update MySQL tables.

**Robot**: Run downloader script to sync card images and update local SQLite database with image hashes.

## Authentication

Backend uses JWT tokens issued on login. Robot software stores tokens via `token_manager.py` and includes in gRPC metadata. Web frontend configures the generated client with JWT token via `setApiToken()` in `web/frontend/src/lib/api.ts`. JWT secret configured via `JWT_KEY` environment variable.

Protected endpoints require `SelectiveAuthInterceptor` middleware (see `backend/internal/server/middleware.go`).

## Web Frontend Client Usage

The TypeScript API client is automatically generated from the backend's OpenAPI spec (which is generated from protobuf definitions).

**Client structure** (`web/frontend/src/generated/api/`):
- `services/` - Service classes for each API (UserServiceService, LibraryServiceService, CardsService)
- `models/` - TypeScript interfaces for request/response types
- `core/OpenAPI.ts` - Client configuration (base URL, auth token)

**Usage pattern**:
1. Configure client on app startup: `configureApiClient(baseUrl, token)` (see `src/lib/api.ts`)
2. Import service classes: `import { LibraryServiceService } from '@/generated/api'`
3. Call methods: `await LibraryServiceService.libraryServiceGetLibraries()`
4. Update token after login: `setApiToken(newToken)`

See `src/lib/example-usage.ts` for detailed examples of common operations.

## Common Gotchas

- **Database signedness**: MySQL foreign keys require exact type matches (e.g., `BIGINT` vs `BIGINT UNSIGNED`)
- **Port conflicts**: Backend runs gRPC on :9090 and HTTP Gateway on :8080 by default
- **Scryfall rate limits**: Bulk data endpoints have no rate limits, but use proper `User-Agent` header
- **Robot deployment**: Uses rsync to specific IP (192.168.88.153) - update Makefile for your setup
- **Generated code**: Always commit generated code (sqlc, protobuf) for reproducible builds
